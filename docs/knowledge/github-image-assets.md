# GitHub に置く画像ファイルの考え方

## 位置づけ

Markdown から参照する画像は、文章だけでは伝わりにくい画面構成、操作対象、状態差分を補助する資料として扱います。

画像は仕様本文の代わりではありません。画面の表示条件、操作、制約、例外はテキストで残し、画像は読み手が全体像をつかむための参考情報として使います。

## 管理対象にしてよい画像

このリポジトリで Git 管理してよい画像は、次の条件を満たすものに限定します。

| 条件 | 内容 |
| --- | --- |
| 用途が明確 | docs の説明を補助する画像である |
| 枚数が限定的 | 主要画面や代表状態に絞る |
| サイズが小さい | 1 ファイルあたり数 MB 未満を目安にする |
| 機密を含まない | token、secret、実 URL、環境値を含まない |
| 差し替え前提ではない | 画面更新のたびに置き換える運用にしない |

画面仕様書に置く場合は、主要画面だけを対象にします。編集モーダル、削除確認、細かい状態差分まで網羅すると、履歴サイズとメンテナンス負荷が増えやすくなります。

## 避ける画像

次の画像は Git 管理しません。

- 実 URL、token、password、client secret、refresh token、Config Vars などが写る画像。
- 個人情報、顧客情報、本番データが写る画像。
- ブラウザのアドレスバーや認証画面を含む画像。
- 一時的な検証結果、PR コメントだけで十分な before / after 画像。
- 画面変更のたびに差し替える前提の画像。
- 自動生成できる大量のスクリーンショット。

秘密情報が写った画像を一度 push すると、通常の削除では Git 履歴から消えません。完全な削除には履歴書き換えが必要になるため、公開前に確認します。

## ファイル名

画像ファイル名は、画面名と状態が分かる小文字 kebab-case にします。

```text
home-default.png
account-list-default.png
account-record-default.png
activity-record-default.png
```

日付、`latest`、`new`、連番だけの名前は避けます。撮影日が必要な場合は、ファイル名ではなく本文側に書きます。

## 配置

docs から参照する画像は `docs/images` に置きます。

```text
docs/images/<screen>-<state>.png
docs/ui/<screen>.md
```

Markdown では相対パスで参照します。

```md
![ホーム画面](../images/home-default.png)
```

画面仕様書では、画像をページ見出しの直下に置くと、読み手が最初に全体像を把握しやすくなります。

## サイズと履歴

GitHub には大きなファイルの制限があります。通常の Git 管理では、単一ファイルが大きくなりすぎると warning や push block の対象になります。

このリポジトリでは、docs 用画像は小さく保ちます。画像を追加する前に、合計サイズと最大ファイルサイズを確認します。

```bash
du -ch docs/images/*
```

画像はバイナリなので、差し替えを繰り返すと Git 履歴に古い実体が残ります。表面上は削除や上書きをしても、過去 commit には残る前提で扱います。

## 圧縮と Git LFS

画像サイズが気になる場合は、まず PNG のまま軽く圧縮することを検討します。PNG のままなら GitHub の Markdown preview で扱いやすく、閲覧環境の差も小さくなります。

Git LFS は、大きいバイナリを Git 本体から分離する仕組みです。ただし、利用者の環境に Git LFS が必要になり、Markdown や PR 上の確認が通常 Git 管理より分かりにくくなる場合があります。

このリポジトリの docs 用スクリーンショットでは、原則として Git LFS は使いません。通常 Git で扱える範囲の小さい画像だけを置きます。

## 代替手段

画像を継続的に増やしたい場合は、Git 管理するスクリーンショットではなく、別の方法を検討します。

| 方法 | 向いていること |
| --- | --- |
| Storybook | コンポーネントや状態差分を最新実装で確認する |
| Playwright などの自動キャプチャ | 必要なときにスクリーンショットを生成する |
| Figma | デザイン意図や画面設計を管理する |
| PR コメント | 一時的な before / after を共有する |

docs には主要画面の補助画像だけを置き、状態差分や網羅的な画面カタログは自動生成や外部ツールへ寄せると、リポジトリを軽く保ちやすくなります。

## 追加前の確認

画像を Git 管理する前に、次を確認します。

- docs の本文だけでは伝わりにくい内容を補っている。
- 主要画面または代表状態に絞られている。
- 実 URL、secret、token、環境値が写っていない。
- 本番データや不要な個人情報が写っていない。
- ファイルサイズが過大ではない。
- 将来の差し替え前提ではなく、参考画像として扱える。
